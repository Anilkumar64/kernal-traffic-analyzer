# 📄 KERNEL TRAFFIC ANALYZER — MASTER PLAN

## 🎯 PROJECT GOAL

Build a **Process-Level Network Observability System (Linux)**

System must answer:

* Which process owns which connections?
* How much data is flowing (per connection & per process)?
* When did connections start/stop?
* Where is traffic going (IP → domain → geo)?
* How does traffic travel (route / hops)?

---

# 🧠 CURRENT STATE (WHERE WE ARE)

### ✅ DONE

* Netfilter-based packet capture
* `/proc/traffic_analyzer` interface
* Flow aggregation (IP + ports)
* Qt GUI (table + bandwidth calculation)
* Stable parsing using `|` delimiter

### ❌ PROBLEMS

* PID mapping is incorrect (`current` used)
* Packet-based view instead of connection-based
* No socket ownership tracking
* No connection lifecycle
* No DNS / Geo / route awareness

👉 System is **observationally weak**

# 🚧 ROADMAP (STRICT PHASES)

---

## 🔵 PHASE 1 — KERNEL CORRECTNESS (CRITICAL)

### Goal:

Accurate **process ↔ connection mapping**

### Tasks:

* Replace:

  ```c
  current   ❌
  ```
* Use:

  ```c
  struct sock *sk = skb_to_full_sk(skb);
  ```
* Extract:

  * socket (`sk`)
  * UID (`sock_i_uid(sk)`)
* Map:

  ```text
  socket → inode → PID
  ```

### Outcome:

```text
REAL process attribution
```

---

## 🟢 PHASE 2 — CONNECTION MODEL

### Goal:

Move from packets → connections

### Track per connection:

* src_ip, dest_ip
* src_port, dest_port
* protocol
* bytes_in / bytes_out
* packets
* first_seen / last_seen

### Add:

* connection identity (5-tuple)

---

## 🟡 PHASE 3 — PROCESS INTELLIGENCE

### Goal:

Understand process behavior

### Per PID:

* total bandwidth
* number of active connections
* protocol distribution

---

## 🔴 PHASE 4 — NETWORK AWARENESS

* DNS resolution:

  ```

  IP → country / city / coordinates
  ```
